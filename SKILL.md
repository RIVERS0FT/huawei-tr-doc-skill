# Huawei TR Design Document Skill

> 用一句话生成华为 TR 风格 / IPD 风格阶段设计文档。
>
> 注意：本 skill 参考公开资料中常见的 TR1-TR6 阶段关注点组织设计文档，不声称属于华为内部官方模板，也不生成“评审报告”。

## 1. 适用场景

当用户希望生成以下内容时，使用本 skill：

- 一句话生成 TR1、TR2、TR3、TR4、TR5、TR6 阶段设计文档
- 根据项目描述生成华为 TR 风格 / IPD 风格设计文档
- 为软件、平台、AI Agent、仿真系统、SaaS、后端服务、数据平台生成阶段性设计文档
- 将粗略需求扩展为可落地、可设计、可评审的 Markdown 设计文档
- 生成 TR1-TR6 全套设计文档包

典型用户输入：

```text
生成 TR1 设计文档：做一个多 Agent 网络攻防仿真平台
```

```text
一句话生成华为 TR3 设计文档：AgentNetworkSimulation 支持 agent 间 HTTP 消息转发、日志追踪和拓扑回放
```

```text
为“AI 自动生成会议纪要系统”生成 TR1-TR6 全套设计文档
```

## 2. 总体原则

1. 默认使用中文输出。
2. 默认输出 Markdown 文档。
3. 输出物是“设计文档”，不是“评审报告”；可以包含“评审准备信息”“设计检查表”“设计结论”，但标题和主体必须围绕设计展开。
4. 如果用户没有指定 TR 阶段，先根据语义判断：
   - 偏想法、立项、概念、可行性：生成 TR1 概念设计文档
   - 偏需求、规格、SRS、范围确认：生成 TR2 需求规格设计文档
   - 偏架构、总体方案、概要设计：生成 TR3 概要设计文档
   - 偏详细设计、模块、接口、数据结构：生成 TR4 详细设计文档
   - 偏测试、集成、验证、样机、试运行：生成 TR5 验证设计文档
   - 偏发布、上线、交付、运维、量产：生成 TR6 发布交付设计文档
   - 用户要求“全套”“完整 TR 文档包”：生成 TR1-TR6 全套设计文档
5. 对用户一句话中没有给出的信息，不要编造成事实；应写入“假设”“待确认”“风险”或“建议补充”。
6. 输出要能直接作为研发设计输入：必须包含设计目标、设计范围、核心方案、关键约束、风险、待确认项、设计检查表和下一步。
7. 不把 TR1 写成详细设计；TR1 重在需求洞察、产品概念、概念方案、可行性和初始风险。
8. 不把 TR2 写成架构设计；TR2 重在需求分解、规格定义、验收口径和追踪关系。
9. TR3 重在总体技术方案、系统架构、概要设计和关键技术路线。
10. TR4 重在详细设计、模块接口、数据结构、异常处理和集成方案。
11. TR5 重在验证设计、测试方案、集成验证、缺陷闭环和质量门禁。
12. TR6 重在发布方案、交付方案、运维方案、回滚方案和生命周期管理。

## 3. 一句话扩展方法

当用户只给一句话时，按下面步骤扩展：

### 3.1 抽取项目卡片

从用户输入中抽取：

| 字段 | 说明 |
|---|---|
| 项目名称 | 如果没有明确名称，从核心名词中生成一个临时名称 |
| 项目类型 | 软件系统、平台、工具、AI Agent、仿真系统、数据服务等 |
| 目标用户 | 业务用户、开发者、管理员、测试人员、运维人员等 |
| 核心目标 | 这个项目解决什么问题 |
| 关键能力 | 主要功能和差异化能力 |
| TR 阶段 | TR1-TR6 或全套 |
| 设计深度 | 概念设计 / 规格设计 / 概要设计 / 详细设计 / 验证设计 / 发布设计 |
| 约束 | 性能、安全、兼容、部署、成本、时间等 |

### 3.2 生成合理假设

只允许生成“显式标注为假设”的内容，例如：

```md
> 假设：系统面向内部研发团队使用，首期以单机或 Docker Compose 部署为主，后续支持 Kubernetes 部署。
```

### 3.3 生成待确认问题

至少列出 5 个待确认问题，覆盖：

- 需求边界
- 用户角色
- 输入输出
- 非功能指标
- 外部依赖
- 安全合规
- 测试验收
- 发布时间

### 3.4 生成风险清单

风险清单必须包含：

| 字段 | 说明 |
|---|---|
| 风险编号 | R-001 |
| 风险描述 | 风险是什么 |
| 影响 | 对范围、质量、成本、进度的影响 |
| 概率 | 高 / 中 / 低 |
| 等级 | 高 / 中 / 低 |
| 应对措施 | 规避、降低、转移、接受 |
| 责任人 | 默认“待定” |
| 状态 | Open / Mitigating / Closed |

## 4. TR 阶段设计文档定义

| 阶段 | 设计文档名称 | 核心设计问题 | 输出重点 |
|---|---|---|---|
| TR1 | 产品概念与可行性设计文档 | 需求是否值得做，概念方案是否可行 | 需求、场景、系统概念、可行性、初始风险 |
| TR2 | 需求分解与规格设计文档 | 需求是否完整，规格是否可实现、可验收 | 需求分解、规格、验收标准、追踪矩阵 |
| TR3 | 总体方案与概要设计文档 | 架构路线是否合理，系统边界是否清晰 | 总体架构、模块划分、关键技术方案 |
| TR4 | 详细设计与模块设计文档 | 设计是否可实现、可编码、可集成 | 模块设计、接口、数据结构、异常处理 |
| TR5 | 集成验证与测试设计文档 | 如何验证设计是否满足需求和质量目标 | 测试方案、验证场景、质量门禁、缺陷闭环 |
| TR6 | 发布交付与运维设计文档 | 如何安全上线、交付、运维和回滚 | 发布计划、运维方案、回滚方案、遗留风险 |

## 5. 输出格式要求

默认输出结构：

```md
# <项目名称> <TRx> <阶段名称>设计文档

## 1. 文档信息
## 2. 一句话输入与需求解析
## 3. 项目背景与目标
## 4. 设计范围与非目标
## 5. 假设与约束
## 6. 本阶段设计内容
## 7. 关键方案设计
## 8. 需求 / 设计 / 验证追踪
## 9. 风险清单
## 10. 待确认问题与闭环计划
## 11. 设计检查表
## 12. 设计结论与下一步
```

要求：

- 表格必须有表头。
- 多行表格内容可使用 `<br>`。
- 文档中的不确定内容必须标记为“待确认”。
- 设计结论可以默认写为“当前设计可进入下一阶段，但需关闭以下问题”。
- 不要输出空章节；如果信息不足，写“当前设计假设”和“待补充”。
- 避免把文档写成会议纪要、评审报告或项目汇报 PPT。

## 6. 针对软件项目的增强内容

如果项目是软件、平台或 AI Agent 系统，额外包含：

- 用户角色和权限模型
- 系统上下文图说明
- 模块划分
- 接口设计
- 数据模型
- 日志与可观测性
- 安全设计
- 部署和运维假设
- 测试策略
- 需求追踪矩阵

如果项目涉及 AgentNetworkSimulation，优先补充：

- agent 角色定义
- message bus / channel / session / trace_id 设计
- Docker 或容器化部署
- 网络事件日志模型
- 拓扑与事件回放
- 异常场景：超时、重试、循环消息、消息风暴、状态不一致

## 7. 模板选择规则

- `templates/tr1.md`：TR1 产品概念与可行性设计文档
- `templates/tr2.md`：TR2 需求分解与规格设计文档
- `templates/tr3.md`：TR3 总体方案与概要设计文档
- `templates/tr4.md`：TR4 详细设计与模块设计文档
- `templates/tr5.md`：TR5 集成验证与测试设计文档
- `templates/tr6.md`：TR6 发布交付与运维设计文档
- `templates/tr-all.md`：TR1-TR6 全套设计文档包

## 8. 质量检查清单

生成完成后，自检：

- [ ] 是否明确了 TR 阶段？
- [ ] 是否避免声称“华为官方模板”？
- [ ] 是否明确输出为“设计文档”而非“评审报告”？
- [ ] 是否从一句话中抽取了项目目标？
- [ ] 是否包含假设和待确认项？
- [ ] 是否包含阶段对应的设计内容？
- [ ] 是否包含风险清单？
- [ ] 是否包含问题闭环计划？
- [ ] 是否包含设计检查表？
- [ ] 是否给出设计结论和下一步？
- [ ] 是否区分了 TR1-TR6 的不同设计深度？
- [ ] 是否适合直接复制为 Markdown 设计文档？

## 9. 输出语气

- 专业、结构化、偏研发设计文档风格。
- 避免营销话术。
- 避免过度承诺。
- 对不确定信息保持审慎。
- 允许在文末给出“建议下一步”。
